0. How to Read This Plan
This is not a feature list with dates attached. This is a build sequence where each week produces a working thing that builds on the previous week's working thing. Nothing is theoretical. Every week ends with something you can run, see, or interact with. If a week's deliverable doesn't work, you don't move forward.
The plan has four phases, each ending with something shippable. You can stop at any phase boundary and have a real product. Phase 1 alone is a business. Phase 2 alone changes how people think about computing. Phase 3 makes it multiplayer. Phase 4 makes it a world.
THE CONSTRAINT
You are one person with a full course load. The plan assumes 15-25 focused hours per week on Dreamworld. Some weeks will be lighter (midterms, assignments). Some will be heavier (reading week, breaks). The plan accounts for this with buffer weeks. Do not skip buffer weeks to get ahead. Use them to polish what you built or rest.

THE RULE
Every component you build must pass the soul check before you move on. The soul check is a single question: would you be proud to show this to a stranger? If the answer is no, keep working on it. If the answer is yes, move forward. Speed comes from not building things twice, and you only avoid rebuilding by getting it right the first time.

1. Technology Decisions (Locked)
These decisions are final. Do not revisit them during development. Revisiting technology choices mid-build is the number one project killer for solo developers.
Frontend
•	Renderer: Three.js (r170+). WebGL2 now, WebGPU renderer path later. Do not use WebGL-specific features directly; always go through Three.js abstractions.
•	Physics (client): Rapier.js (WASM). Deterministic, fast, active development. Not Cannon.js (abandoned). Not Ammo.js (bloated).
•	UI framework: Vanilla HTML/CSS/JS for the in-game computer overlay. No React in the game client. React adds build complexity you don't need. Use it later for standalone tools if needed.
•	State: Custom event bus. Packets are the state management system. No Redux, no Zustand. The Packet system IS state management.
•	Local storage: IndexedDB (via idb library) for the personal knowledge graph. 50-200MB capacity is more than enough for years of braindumps.
Backend
•	Gateway: Node.js 20+ with the ws library (not socket.io). Raw WebSocket gives you control over the Packet format. socket.io adds abstraction you'll fight against.
•	World processes: Node.js child processes via child_process.fork(). Same language as gateway, shared Packet validation code.
•	Agent service: Python 3.11+ with FastAPI. Async endpoints for non-blocking LLM calls. Anthropic SDK as primary, OpenAI as fallback.
•	Knowledge service: Python 3.11+ with FastAPI. Postgres queries, embedding generation, semantic search.
•	Database: PostgreSQL 16 with PostGIS extension for spatial queries. pgvector extension for embedding similarity search.
•	Server: Hetzner Cloud. Start with a CX41 (4 vCPU, 16GB RAM, ~$16/mo). Upgrade when needed.
Infrastructure
•	Process management: PM2 for keeping gateway and Python services alive. World processes managed by gateway directly.
•	TLS: Let's Encrypt via Certbot. Auto-renewal. WSS:// and HTTPS:// on all endpoints.
•	Reverse proxy: Nginx. Routes HTTPS to gateway, API paths to Python services, static assets from disk.
•	Backups: pg_dump cron job to Hetzner Storage Box (separate from server). Daily full, hourly WAL archiving.
•	Monitoring: Simple: PM2 monitoring + custom health check endpoint + Discord webhook alerts on crash.
Tooling
•	Version control: Git + GitHub. Monorepo with /client, /gateway, /services/agent, /services/knowledge, /shared (Packet types).
•	Package management: npm for JS, pip with requirements.txt for Python.
•	Development: VS Code. No Docker in development (adds complexity). Docker for production deployment later.

6. Phase 4: The 3D Internet (Post-Launch)
No fixed timeline. These are expansions built based on user demand, revenue, and your capacity. Each is independently shippable.

Expansion: Visual Programming System (The Eyes)
Code rendered as walkable 3D space. For loops as spirals, functions as portals, variables as containers. The most technically ambitious feature. Build only when the core platform is stable and generating revenue. Estimate: 6-8 weeks dedicated.

Expansion: Camera Recording System
Record game state as Packet streams, not pixels. Replay from any angle. Edit after the fact. Live streaming. The Camera organ from the PRD. Estimate: 3-4 weeks.

Expansion: Voice Chat (WebRTC)
Proximity-based spatial audio. Hear players get louder as they approach. Directional audio in 3D. WebRTC peer-to-peer with server-mediated signaling. Estimate: 4-5 weeks.

Expansion: Working Phones
Inventory item with small screen interface. Text messaging, contact lists, notifications. Same screen-object architecture as monitors but handheld. Estimate: 2-3 weeks.

Expansion: Image Upload with AI Moderation
Upload endpoint routes through classification API before image becomes available as texture. OpenAI moderation endpoint or Google Cloud Vision SafeSearch. Never stored until approved. Estimate: 2 weeks.

Expansion: Deep Memory Meta-Analysis
The Knowledge Graph learns from itself. Pattern extraction across all episodic data. Agent config optimization. Predictive suggestions. This is the brain getting smarter over time. Estimate: ongoing.

Expansion: Mobile Native App
React Native or Flutter wrapper around the Three.js client. Reduced polygon budgets, simplified shaders, touch controls. Major undertaking. Estimate: 8-12 weeks.

Expansion: WebGPU Renderer
Swap Three.js renderer from WebGL to WebGPU. Significant performance improvement. Compute shaders for client-side physics. Estimate: 2-3 weeks if Three.js WebGPU renderer is stable.

Expansion: Multi-Agent Architecture (Moltbot Engine)
Inspired by the OpenClaw multi-agent work: agents as entities with components (cognition, memory, tools, personality, attention, intent). Game-engine-style tick loop for agent processing. State machine (IDLE, REASONING, EXECUTING, WAITING, ERROR). Event bus for decoupled agent-to-agent messaging. Hot-swappable personality modules. This turns the agent system from a request/response service into a living cognitive architecture. Estimate: 4-6 weeks.

Expansion: Enterprise + Education Licensing
Private instances for schools and companies. Custom branding, SSO integration, admin dashboards, usage analytics, SLA support contracts. Revenue diversification. Estimate: 4-6 weeks of product work + ongoing support.

7. Risk Register

Risk: Scope creep kills momentum
Mitigation: Every week has a checklist. If it's not on the checklist, it doesn't get built that week. New ideas go in a backlog document, not into the current sprint. Review the backlog monthly, not daily.
Risk: LLM API costs exceed revenue
Mitigation: Aggressive caching of agent outputs. Identical or near-identical requests return cached results. Token limits per user per day. Monitor costs weekly. Switch to cheaper models (Haiku) for simple tasks, reserve Opus/Sonnet for complex reasoning.
Risk: LLM API downtime breaks the platform
Mitigation: Fallback chain (Anthropic -> OpenAI -> local model for basic tasks). Agent service queues tasks during outage, processes when API returns. Graceful degradation: if agents are down, the 3D world still works for building, exploring, and multiplayer.
Risk: Solo developer burnout
Mitigation: The Soul section exists for a reason. Sleep schedule locked. No triple all-nighters. Buffer weeks are not optional. If a week is missed, the plan shifts; it doesn't compress. Bolivia by end of year is non-negotiable — the plan accounts for travel.
Risk: Nobody cares about the alpha
Mitigation: The Telegram bot (Phase 0) validates demand before you build the 3D world. If nobody uses the bot, reconsider scope before investing 3 months in the house. The bot is your market test.
Risk: Multiplayer is buggy and unfun
Mitigation: Multiplayer ships in beta (Phase 2), not alpha. Alpha is single-player and polished. This means the core experience (house + computer + agent) is solid before multiplayer complexity enters. Multiplayer bugs don't poison first impressions.
Risk: Security breach before proper security
Mitigation: Beta warning on all surfaces. No image uploads. No sensitive data storage. Passwords hashed. HTTPS everywhere. DB not public. The warning is both honest and legally protective. Security hardens incrementally with each phase.
